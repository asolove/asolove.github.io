The virtual DOM is not a deep mystery. It is a small algorithm with a few fiddly details.

To many people, the virtual DOM and component model seem like mysteries that mere mortals shouldn’t try to understand. As I hopefully demonstrated, this isn’t true. It’s a reasonably simple algorithm with a few fiddly details. While React is implemented somewhat differently, knowing Preact’s internals is still a good first mental model for approaching it.

§1.2

The core diff loop

The heart of the renderer is roughly this — a recursive walk that reconciles a virtual node against the DOM node it last produced, returning the (possibly new) DOM node to its parent.[2]

[2] Simplified for clarity. The real diff in Preact 8 also threads context, an SVG flag, and a hydration mode through every call.
Listing 1. preact / src / diff.js 
// Diff a vnode against an existing dom node.
function diff(dom, vnode, parent) {
  if (typeof vnode === 'string') {
    return diffText(dom, vnode);
  }
  if (typeof vnode.nodeName === 'function') {
    return buildComponentFromVNode(dom, vnode);
  }
  return innerDiffNode(dom, vnode);
}
Simplified from preact 8.x · 9 lines

Everything else in the codebase is in service of those three branches: handling text nodes, building up component instances, and walking the children of an element. Once you can hold those branches in your head, the rest is bookkeeping.■
